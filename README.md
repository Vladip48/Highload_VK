## 1. Тема и целевая аудитория

### 1.1 Тема

Яндекс.Такси - сервис мобильного заказа такси, входящий в экосистему Яндекс Go. Позволяет пользователям находить ближайших водителей, отслеживать поездку в реальном времени и оплачивать поездки через приложение. Сервис работает в России, странах СНГ и ряде других регионов.

### 1.2 Функционал MVP

- Авторизация пользователя
- Отправка запроса на вызов такси
- Поиск ближайших свободных водителей  
- Назначение заказа одному из водителей  
- Отслеживание поездки в реальном времени 
- Расчёт стоимости поездки и оплата
- Рейтинг водителей

### 1.3 Ключевые продуктовые решения
- Surge Pricing - динамическое ценообразование [18, 21]
- Динамическое размещение водителей по заказам [20]
- Двусторонняя система рейтингов (водители тоже оценивают пассажиров) [19]

### 1.4 Целевая аудитория

- **География:** Россия, Казахстан, Армения, Грузия, Узбекистан  
- **Размер аудитории:**  
  - Месячная аудитория - 53,2 млн пользователей [1]
  - Дневная аудитория - 14 млн пользователей  

- **Средний возраст пользователей:** 18-29 лет [16]  
- **Устройства:** 90% - мобильные, 10% - десктоп [17]

## 4. Локальная балансировка нагрузки

#### 4.1 Схема балансировки

Выбрана двухуровневая архитектура L4 + L7. 

| Уровень | Реализация | Описание работы | Механизм резервирования |
|---------|----------|----------|----------|
| **L4-балансировка** | LVS (Linux Virtual Server)| Входящий трафик распределяется между узлами L7 методом Direct Routing, обеспечивая высокую производительность. Исходящий трафик идёт напрямую к клиенту, что минимизирует нагрузку на балансировщик | Схема N×2, keepalived мониторит доступность нод, VRRP обеспечивает автоматический failover Virtual IP на резервный узел|
| **L7-балансировка** | Nginx (HTTP Reverse Proxy) | SSL Termination, распределение запросов по микросервисам | Active-Active N+1, при сбое одного узла нагрузка равномерно распределяется между оставшимися, политика failover (proxy_next_upstream) для перенаправления запросов при сбоях бэкендов |

#### 4.2 Расчёт количества балансировщиков

Исходные данные 

| Параметр | Значение |
|---------|----------|
| **Пиковый RPS** | 236 480 запросов в секунду |
| **Пиковый трафик** | 0,8 Гбит/с |

Узлы на базе с 8 CPU и сетевой картой 10GbE [32]
Ограничителем для L4 выступает только пропускная способность сетевых интерфейсов, так как SSL-терминация не выполняется. 
*   По пропускной способности: 0,8 Гбит/с / 10 Гбит/с = 0,08 → 1 сервер
*   Применяем формулу резервирования N * 2: 1 * 2 = 2 сервера на ДЦ

Ограничителями для L7 выступают терминация SSL и пропускная способность сети.
Интенсивность новых TLS-соединений принята равной равной 50% общего RPS:
*   По ограничению SSL: 236 480 СPS * 0,5 / 6 676 CPS = 17,7 → 18 серверов 
*   По пропускной способности: 0,8 Гбит/с / 10 Гбит/с = 0,08 → 1 сервер
*   Применяем формулу резервирования N + 1: 18 + 1 = 19 сервера NGINX на ДЦ

| Уровень | Количество | Конфигурация |
|---------|----------|----------|
| **L4** | 2 | CPU 8 Cores, NIC 10GbE |
| **L7** | 19 | CPU 16 Cores, NIC 10GbE |

## 6. Физическая схема БД

### Выбор СУБД для каждой таблицы
| Название таблицы | СУБД | Обоснование выбора |
|-----------------|----------------------|------------------|
| **User, Passenger, Driver, Driving_license, Address, Passenger_Address, Payment_credit, Passenger_Payment_credit, Trip, Rating, Payments** | PostgreSQL | OLTP нагрузка, транзакционная целостность |
| **Passenger_geoposition, Driver_geoposition, Surge_pricing, Hexagon** | Redis | Низкая задержка, частые обновления |
| **Trip_audit_log, Rating_history** | Clickhouse | Аналитика, логгирование |

### Индексы
| Таблица | Поле | Тип индекса | Обоснование |
|---------|------|-------------|-------------|
| **User** | `id` | B-tree | Поиск пользователя по ID для связи с другими таблицами |
| **User** | `phone_number` | B-tree | Поиск пользователя при авторизации по номеру телефона |
| **Driver** | `status` | B-tree | Фильтрация водителей по статусу для поиска свободных |
| **Driver** | `current_location` | GiST | Поиск ближайших водителей к точке заказа по координатам |
| **Trip** | `passenger_id` | B-tree | Получение истории поездок конкретного пассажира |
| **Trip** | `driver_id` | B-tree | Получение истории поездок конкретного водителя |
| **Trip** | `status`, `created_at` | B-tree | Выборка активных поездок с сортировкой по времени создания |
| **Address** | `location` | GiST | Поиск адресов в радиусе от координат и расчёт расстояний |
| **Payments** | `trip_id` | B-tree | Поиск платежа по идентификатору поездки для проверки оплаты |
| **Driver_geoposition** | `driver_id` | Redis key | Получение текущих координат водителя для отслеживания на карте |
| **Driver_geoposition** | `hexagon_id` | Redis key | Поиск всех водителей в зоне шестиугольника для расчёта surge pricing |
| **Passenger_geoposition** | `passenger_id`, `trip_id` | Redis key | Отслеживание местоположения пассажира во время активной поездки |
| **Surge_pricing** | `zone_id` | Redis key | Получение коэффициента спроса для расчёта стоимости заказа в зоне |
| **Hexagon** | `hexagon_id` | Redis key | Получение границ гео-зоны для визуализации карты спроса |
| **Trip_audit_log** | `trip_id` | B-tree | Просмотр истории изменений конкретной поездки для аудита |
| **Trip_audit_log** | `time` | B-tree | Фильтрация логов по временному диапазону для отчётности и очистки |
| **Rating_history** | `rated_user_id` | B-tree | Расчёт среднего рейтинга пользователя на основе истории оценок |
| **Rating_history** | `time` | B-tree | Анализ динамики рейтинга пользователя за период времени |

### Денормализация
| Таблица | Поле | Обоснование |
|---------|------|-------------|
| **Driver** | `rating_avg` | Исключает JOIN с таблицей `Rating` при отображении карточки водителя |
| **Driver** | `total_trips` | Исключает агрегацию и JOIN с таблицей `Trip` для подсчёта поездок |
| **Trip** | `surge_multiplier` | Исключает JOIN с таблицей `Surge_pricing` при расчёте итоговой стоимости |
| **User** | `rating` | Исключает JOIN с таблицей `Rating` при загрузке профиля пользователя |
